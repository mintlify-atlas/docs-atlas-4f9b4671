---
title: "POST /api/v1/permission/assign — Assign a Permission to a Role"
sidebarTitle: "Assign Permission"
description: "Links a permission to a role, granting access to that route for all users with that role. Requires JWT and the POST /api/v1/permission/assign permission."
---

Creates an entry in the `permissionXRole` pivot table that links a `permissionId` to a `roleId`. Once assigned, all users whose role matches `roleId` are granted access to the route identified by that permission. The assignment takes effect immediately — subsequent requests from users with that role will pass the `roles.middleware` check for the associated route.

The underlying `INSERT IGNORE` statement means that if the assignment already exists, no error is raised and the operation is silently treated as a no-op at the database level. However, if the combination is truly new, a `data:changed` Socket.IO event is broadcast to all connected clients.

## Authentication

Requires a valid JWT **and** the `POST /api/v1/permission/assign` permission assigned to the caller's role.

## Request

**Method:** `POST`
**Path:** `/api/v1/permission/assign`

### Headers

<ParamField header="Authorization" type="string" required>
  Bearer token obtained from the login endpoint. Format: `Bearer <token>`
</ParamField>

<ParamField header="Content-Type" type="string" required>
  Must be `application/json`.
</ParamField>

### Body

<ParamField body="roleId" type="number" required>
  The numeric ID of the role to assign the permission to. Must be a JSON number (not a string).
</ParamField>

<ParamField body="permissionId" type="number" required>
  The numeric ID of the permission to assign. Must be a JSON number (not a string). Retrieve this from [List Permissions](/api/permissions/list) or [Get by URI](/api/permissions/get-by-uri).
</ParamField>

```json Example Request Body
{
  "roleId": 2,
  "permissionId": 5
}
```

## Response

### 200 OK

Returned when the assignment is successfully created (or silently ignored as a duplicate).

<ResponseField name="success" type="boolean">
  Always `true` for successful responses.
</ResponseField>

<ResponseField name="message" type="string">
  Human-readable confirmation message.
</ResponseField>

<ResponseField name="data" type="null">
  Always `null` for this operation.
</ResponseField>

```json Example Response
{
  "success": true,
  "message": "Permiso asignado al rol exitosamente",
  "data": null
}
```

### Error Responses

| Status | Description |
|--------|-------------|
| `400 Bad Request` | `roleId` or `permissionId` is missing or not a JSON number type. |
| `401 Unauthorized` | Missing or invalid JWT token. |
| `403 Forbidden` | Authenticated user's role lacks the `POST /api/v1/permission/assign` permission. |
| `409 Conflict` | Foreign-key violation — the specified `roleId` or `permissionId` does not exist. |

```json 400 Bad Request — Wrong Type
{
  "success": false,
  "message": "Solicitud inválida",
  "data": null
}
```

## Code Example

```bash cURL
curl -X POST http://localhost:3000/api/v1/permission/assign \
  -H "Authorization: Bearer <your_token>" \
  -H "Content-Type: application/json" \
  -d '{
    "roleId": 2,
    "permissionId": 5
  }'
```

## Socket.IO Events

On success, the server emits the following events:

| Event | Target | Operation | Status |
|-------|--------|-----------|--------|
| `operation:progress` | Requesting socket | `permissions:assign` | `start` → `success` |
| `data:changed` | All clients | `permissions:assign` | Broadcast (no payload data) |
